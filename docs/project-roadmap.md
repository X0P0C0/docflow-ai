## 1. 这份文档是干什么的

这份文件可以理解成你接下来做项目的长期笔记。

它的作用是：

- 记录我们已经确定下来的项目方向
- 记录后面还要做的事情，防止遗忘
- 用尽量容易懂的话解释重要术语
- 帮你把一个项目真正做成能用于求职和面试的代表作

## 2. 当前推荐的项目方向

推荐主项目：

`AI 知识库 + 工单协同平台`

项目暂定名：

`DocFlow AI`

一句话介绍：

一个面向团队协作的系统，支持知识管理、工单处理，以及 AI 辅助办公。

为什么这个方向适合你现在做：

- 它看起来像真实公司会用的产品，不像纯练手 demo
- 它天然适合做前后端分离
- 它可以同时覆盖 Java、数据库设计、Redis、缓存、权限、AI 功能这些面试高频点
- 它既能展示后端能力，也能展示前端页面能力
- UI 还可以参考 `awesome-design-md` 里的成熟产品风格，做出来更像真正的 SaaS 产品

## 3. 我们现在真正要追求的目标

不是为了“做很多个项目”。

而是为了做出这样一个东西：

- `一个能代表你水平的旗舰项目`
- `一个面试时能深入讲明白的项目`
- `一个既能提升代码能力，也能提升工程眼界的项目`

当前建议：

- 主项目先做 `1 个`
- 如果后面时间还够，再补 `1 个` 小型专项项目

补充策略说明：

如果后面还有时间，再补 `1 个小而专的项目`，专门强化 `高并发` 或某个专项能力。

为什么不建议一开始就做很多项目：

- 项目一多，很容易每个都做浅
- 做浅了，面试时反而讲不出深度
- 一个完整而扎实的项目，通常比三个半成品更有竞争力

## 5. UI 风格方向

结合 `awesome-design-md`，推荐这样参考：

- 工作台 / 控制台：参考 `Linear`
- 知识库页面：参考 `Notion` 或 `Mintlify`
- 官网或项目展示页：参考 `Stripe` 或 `Apple`
- AI 助手区域：参考 `Claude` 或 `VoltAgent`

我们要的视觉目标：

- 干净，但不能单调
- 像成熟产品，不像“AI 一把梭生成的模板页”
- 信息层级清晰
- 版式有节奏感
- 细节统一，有质感

要记住一点：

好看的 UI 不只是颜色好看，还包括：

- 留白
- 字体层级
- 间距统一
- 卡片布局节奏
- 空状态设计
- 按钮和交互反馈
- 页面整体一致性

## 8. 重点术语解释

这一部分是给现在的你看的，不要求一次全懂，先有个概念就行。

### 8.1 前后端分离

前端负责页面、交互、展示。
后端负责业务逻辑、数据处理、接口。
两边通过接口通信。

为什么公司喜欢这么做：

- 职责更清晰
- 前后端能并行开发
- 后期维护和扩展更方便

### 8.2 RBAC

RBAC 的全称是 `基于角色的权限控制`。

你可以这样理解：

- 管理员能管理所有用户
- 客服能处理工单
- 普通员工只能提交和查看自己的工单

核心思路不是“直接给每个人分配权限”，而是：

- 先定义角色
- 给角色分配权限
- 再把角色分给用户

这样管理起来更清晰。

### 8.3 Redis

Redis 是一种非常快的内存数据库。

常见用途：

- 缓存热点数据
- 保存登录信息
- 保存验证码
- 做计数器
- 做限流

为什么重要：

如果每次请求都直接查 MySQL，用户一多，数据库压力会很大。
Redis 就像一个“高速缓冲区”。

### 8.4 缓存

缓存就是把常用数据先放到更快的地方，下次再访问时不用每次都查数据库。

例如：

一篇热门知识文章被看了很多次，如果每次都查数据库就很浪费。
可以先把它放到 Redis 里，这样访问速度更快。

### 8.5 缓存穿透

缓存穿透通常指：

大量请求访问的是“根本不存在的数据”。

例如：

有人一直查一个不存在的文章 ID。
因为缓存里没有，数据库里也没有，所以每次都要打到数据库。

这样数据库就会被白白消耗性能。

常见解决方案：

- 给不存在的数据也缓存一个空值，短时间内别再去查数据库
- 更进一步可以用布隆过滤器

### 8.6 缓存击穿

缓存击穿通常指：

某个“特别热门”的数据刚好过期了，结果一瞬间很多请求同时去查数据库。

例如：

一篇非常热门的知识文章缓存刚过期，成千上万的人同时访问它。

常见解决方案：

- 加互斥锁
- 用逻辑过期方案

### 8.7 缓存雪崩

缓存雪崩通常指：

很多缓存数据在差不多同一时间一起过期，导致大量请求一下子都打到数据库。

常见解决方案：

- 给过期时间加随机值，避免同时失效
- 做降级处理
- 做流量保护

### 8.8 高并发

高并发就是：

同一时间有很多用户一起访问系统。

比如：

- 很多人同时搜索知识库
- 很多人同时提交工单
- 很多人同时打开首页看板

面试官关心的是：

- 你的系统哪里会成为瓶颈
- 数据库怎么保护
- Redis 怎么帮忙
- 你的方案在压力下稳不稳

### 8.9 幂等性

幂等性可以简单理解为：

同一个请求重复执行多次，最终结果也不应该乱掉。

比如：

用户点了两次“提交工单”，系统不能生成两条一样的工单。

### 8.10 异步处理

异步处理就是：

有些操作不用在当前请求里立刻做完，可以先返回结果，再慢慢处理。

比如：

- 发通知
- 生成 AI 总结
- 更新搜索索引

这样做的好处：

- 用户不用等太久
- 系统高峰期压力更小

### 8.11 RAG

RAG 一般指：

`检索增强生成`

简单理解：

先去找相关资料，再让 AI 基于这些资料回答问题。

为什么它对这个项目有用：

因为我们的 AI 助手可以优先参考你系统里的知识库内容，而不是胡乱编答案。

### 8.12 SLA

SLA 可以理解为：

服务时效要求或服务承诺。

例如：

- 高优先级工单必须在 30 分钟内响应
- 普通工单要在 24 小时内处理

有了 SLA，工单系统就会更像真实企业产品。

## 10. 什么是 MVP

MVP 的意思是：

`最小可用产品`

简单理解就是：

先做出一个“虽然不大，但已经完整、能演示、能运行”的版本。

当前建议的 MVP 范围：

- 登录
- 角色权限控制
- 知识文章的新增、编辑、查看
- 工单的创建、分配、更新
- 搜索
- 数据看板
- 1 到 2 个真正有用的 AI 功能

一定不要一开始就追求“功能全做完”。
先把最核心的闭环做出来，才是正确路线。
