# DocFlow AI 项目立项文档

## 1. 项目基本信息

### 1.1 项目名称

`DocFlow AI`

### 1.2 项目定位

这是一个面向团队内部协作的智能知识库与工单协同平台。

它的目标不是做一个普通的后台管理系统，而是做一个：

- 具备真实企业业务味道的 SaaS 风格产品
- 支持前后端分离开发的全栈项目
- 能覆盖 Java 面试高频知识点的代表作
- 能接入 AI 能力并讲清楚落地场景的项目

### 1.3 项目目标

这个项目最终要同时服务 3 个目标：

- 求职作品集展示
- 全栈实战能力提升
- 面试表达素材沉淀

---

## 2. 为什么选这个项目

相比纯管理后台、博客系统、商城系统，这个项目更适合你现在的阶段。

原因如下：

- 它比普通 CRUD 项目更像真实企业产品
- 它可以自然覆盖权限、缓存、搜索、协作、通知、AI 等能力
- 它的业务复杂度适中，不会空泛，也不会一开始就过重
- 它很适合在面试中展开讲业务场景和系统设计
- 它的 UI 更容易做出高级感

## 4. 核心业务场景

为了让这个项目更像真实产品，我们先设定一个清晰业务背景：

某团队内部有大量常见问题、操作文档和问题反馈。

目前存在的问题：

- 文档分散，搜索效率低
- 工单处理流程不清晰
- 重复问题很多
- 新人很难快速找到解决方案
- 工单回复依赖老员工经验

所以系统要解决这些问题：

- 统一沉淀知识内容
- 规范工单流转过程
- 提供搜索能力
- 用 AI 辅助总结、归类、推荐答案
- 提高团队协作效率

## 5. 技术栈最终建议

这次项目建议采用以下方案。

### 5.1 前端

- `Vue 3`
- `TypeScript`
- `Vite`
- `Pinia`
- `Vue Router`
- `Axios`
- `UnoCSS` 或 `Tailwind CSS` 二选一
- `ECharts`（用于图表看板）

为什么前端推荐 Vue 3：

- 你已经接触过 Vue 3，上手成本更低
- 更容易把精力放在完整项目实现上
- 对当前求职目标来说，先把项目做深比切 React 更重要

### 5.2 后端

- `Java 17`
- `Spring Boot`
- `Spring Security`
- `MyBatis-Plus`
- `MySQL 8`
- `Redis`
- `JWT`
- `Maven`

### 5.3 工程与部署

- `Git`
- `Swagger / OpenAPI`
- `Docker`
- `Nginx`
- `Apifox` 或 `Postman`

### 5.4 AI 能力接入

第一阶段先采用：

- 大模型 API 调用
- Prompt 模板设计

后期再考虑：

- RAG
- 多轮上下文
- AI Agent 工作流

## 6. 第一阶段功能边界

这个项目不能一开始就做太大，所以我们先做 MVP。

MVP 的意思是：

`最小可用版本`

也就是虽然功能不算很多，但已经是一个完整、可以运行、可以演示的系统。

### 6.1 MVP 必做功能

- 角色权限控制
- 知识库文章列表
- 知识库文章详情

- 工单列表
- 工单详情
- 新建工单
- 分配工单
- 更新工单状态
- 搜索功能
- 首页数据看板
- AI 文章总结
- AI 工单回复建议

- 复杂工作流引擎

## 8. 业务流程初稿

### 8.1 知识库主流程

1. 管理员或支持人员创建知识文章
2. 系统保存文章内容、分类、标签
3. 普通用户通过列表或搜索查看文章
4. AI 可以对文章做摘要，帮助快速阅读

### 8.2 工单主流程

1. 普通用户提交工单
2. 系统记录工单内容、优先级、状态
3. 支持人员接单或管理员分配工单
4. 支持人员处理并回复工单
5. AI 根据工单内容给出回复建议或推荐相关文章
6. 工单处理完成后关闭

### 8.3 AI 辅助流程

1. 用户查看知识文章时，AI 生成摘要
2. 用户提交工单后，AI 自动分析问题类型
3. 支持人员处理工单时，AI 推荐相似问题和答复模板

## 12. 一个重要策略提醒

当前最优策略不是“同时做很多个项目”。

而是：

- 先做 `1 个旗舰主项目`
- 如果后面还有时间，再补 `1 个小而专的项目`，专门强化 `高并发` 或某个专项能力

这样做的好处是：

- 主项目负责展示完整能力
- 小项目负责补强面试专项
- 两者组合起来，比多个浅项目更有竞争力
