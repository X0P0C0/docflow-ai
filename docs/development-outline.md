# DocFlow AI 项目开发总大纲

## 1. 这份文档的定位

这份文档不是普通备注，也不是简单待办清单。

它是 `DocFlow AI` 项目的：

- 开发总路线图
- 多轮对话协作大纲
- AI 接力开发执行标准
- 后续实现深度的统一说明书

目标是解决两个问题：

1. 后续新对话应该做什么
2. 后续新对话应该做到什么程度

一句话理解：

`这份文档负责把项目从“能做”推进到“做得专业、做得完整、做得能拿去面试”。`

---

## 2. 项目最高目标

`DocFlow AI` 最终不是要做一个好看的演示壳子，而是要做成一个：

- 可用于求职展示的旗舰项目
- 可覆盖 Java 面试高频知识点的全栈项目
- 具备真实业务气质的 SaaS 风格系统
- 具备持续扩展能力的工程化作品

它最终要同时承载 5 个目标：

1. 简历项目展示
2. 全栈代码能力训练
3. 架构设计与工程化训练
4. 面试表达材料沉淀
5. AI 辅助业务系统落地实践

---

## 3. 项目核心产品定位

项目名称：

`DocFlow AI`

项目方向：

`AI 知识库 + 工单协同平台`

核心业务价值：

- 沉淀知识内容
- 规范工单流程
- 提高搜索效率
- 让 AI 真正参与协作流程
- 提升团队问题处理效率

它不是：

- 普通后台模板
- 单纯博客系统
- 只有聊天框的 AI 外壳
- 纯 CRUD 练手项目

---

## 4. 项目必须长期坚持的原则

后续无论谁接手开发，都必须坚持这些原则。

### 4.1 不推翻现有项目方向

不允许中途把项目改成：

- 商城
- 博客
- OA
- 纯 AI 助手
- 别的技术栈重做

### 4.2 不做“好看但空”的页面

页面必须逐步从：

- 视觉原型
- mock 页面
- 半联调页面
- 真正联调页面

逐步推进，而不是长期停留在演示态。

### 4.3 不只堆功能，要做闭环

后续每做一个模块，都尽量做到：

- 列表页
- 详情页
- 基础数据流
- 路由跳转
- 接口联调

至少形成一个最小闭环。

### 4.4 不写自己讲不清楚的“高级词”

后续如果引入：

- Redis
- 缓存穿透
- 缓存击穿
- 缓存雪崩
- JWT
- 权限控制
- AI 摘要
- RAG

都必须能结合本项目实际业务讲清楚。

### 4.5 所有阶段都要服务面试价值

每一轮开发都要问自己：

- 这部分能不能成为面试素材
- 这部分能不能体现工程思维
- 这部分能不能体现真实业务场景

## 7. 开发阶段总规划

后续开发必须按阶段推进，而不是想到哪做哪。

### 第 1 阶段：基础质量收口

目标：

把当前已完成的内容从“能跑”收口到“稳定、可继续扩展”。

必须完成：

- 修复明显编码问题
- 修复明显交互问题
- 收口登录主线
- 收口 token 恢复逻辑
- 整理文档体系

阶段完成标志：

- 前后端能稳定启动
- 登录后能稳定进入工作台
- 页面没有明显乱码和低级问题

### 第 2 阶段：登录与权限系统完善

目标：

把“有登录”推进到“像真实系统一样登录”。

必须完成：

- 登录页优化
- `/api/auth/login`
- `/api/auth/me`
- JWT 校验
- 路由守卫
- 退出登录
- 当前用户展示
- 基础角色信息展示
- 权限菜单初步控制

阶段完成标志：

- 用户登录后能看到正确身份信息
- 未登录不能进入受保护页面
- 不同角色至少能体现部分权限差异

### 第 3 阶段：知识库主线完善

目标：

把知识库从“有详情页”推进成完整业务线。

必须完成：

- 知识文章列表页独立化
- 文章搜索
- 文章分类筛选
- 文章详情阅读体验优化
- 新建文章页
- 编辑文章页
- 文章发布状态管理

进阶可做：

- 标签管理
- 版本历史
- 浏览量统计

阶段完成标志：

- 知识库具备“列表 -> 详情 -> 编辑/发布”的完整主线

### 第 4 阶段：工单主线完善

目标：

把工单系统从“基础展示”推进成更像真实协作流程。

必须完成：

- 工单列表筛选
- 工单详情页细化
- 工单评论区
- 工单处理记录区
- 工单状态流转
- 指派处理人
- 新建工单页

进阶可做：

- 附件上传
- SLA 提示
- 处理时长统计

阶段完成标志：

- 工单具备“创建 -> 分配 -> 处理 -> 查看记录”的完整业务闭环

### 第 5 阶段：用户与系统管理模块

目标：

补齐系统化后台能力。

必须完成：

- 用户列表
- 用户详情
- 角色列表
- 角色权限展示
- 基础系统管理页

阶段完成标志：

- 项目从功能页升级成“有系统管理能力的产品”

### 第 6 阶段：Redis 与缓存场景落地

目标：

把项目拉到真正适合面试展开的层次。

必须完成：

- Redis 配置落地
- 热门知识文章缓存
- 首页统计数据缓存
- 登录态缓存设计说明
- 缓存穿透场景说明
- 缓存击穿场景说明
- 缓存雪崩场景说明

阶段完成标志：

- 至少有 2 到 3 个真实 Redis 使用点
- 能结合业务讲缓存问题与解决方案

### 第 7 阶段：AI 能力落地

目标：

让 AI 不只是装饰，而是真正服务业务。

优先级最高的 AI 功能：

1. 文章摘要
2. 工单回复建议
3. 工单关联知识推荐

后续进阶：

- 工单自动分类
- 基于知识库的问答
- 简单 RAG

阶段完成标志：

- AI 至少真实参与 2 条业务链路
- AI 功能可被面试时清楚解释

### 第 8 阶段：部署与工程化

目标：

让项目从“本地开发项目”升级成“可展示作品”。

必须完成：

- 前端部署方案
- 后端部署方案
- 数据库与 Redis 部署说明
- 环境配置说明
- Docker / Nginx 基础接入

阶段完成标志：

- 项目具备可部署能力

### 第 9 阶段：面试表达与最终包装

目标：

把项目真正变成求职武器。

必须完成：

- 项目介绍稿
- 技术亮点稿
- Redis 场景讲解稿
- 权限系统讲解稿
- 工单主线讲解稿
- AI 落地讲解稿

阶段完成标志：

- 能把项目完整讲 3 分钟 / 10 分钟 / 30 分钟三个版本

## 8. 当前最优先级的下一步

如果只看现在，最优先顺序是：

1. 收口当前登录与权限骨架
2. 收口工单评论写入、状态流转、指派处理人、新建工单、列表筛选闭环
3. 再回到知识库搜索、筛选、编辑发布
4. 开始 Redis 真实场景接入

当前最推荐的一个主任务：

`开始做用户信息展示与菜单权限`

为什么是它：

- 它已经从“能看”推进到了“能写、能指派、能创建、能筛选”，最适合切入权限和角色差异
- 它天然适合承载真实面试场景，比如状态流转、指派、协作留痕、幂等与通知
- 它完成闭环后，项目的产品感和工程感都会明显更强
- 它后续也适合继续接 Redis、通知、AI 回复建议和知识推荐

## 10. 什么程度算第一阶段结束

`DocFlow AI` 不需要等所有设想全部实现，才算阶段性完成。

当前定义的 `V1 完整版结束标准` 是：

1. 登录、JWT、当前用户恢复、基础权限控制跑通
2. 知识库具备列表、详情，以及至少一项高价值增强能力，例如搜索或分类筛选
3. 工单主线具备 `创建 -> 指派 -> 状态流转 -> 评论留痕 -> 列表筛选`
4. Redis 至少落地 2 个真实业务场景，并且能结合本项目讲清缓存问题
5. AI 至少真实参与 1 到 2 条业务链路，例如文章摘要或工单回复建议
6. 项目具备基础文档、架构说明、部署说明和面试表达材料

满足以上标准后，就可以把项目视为：

- 可以用于求职展示的第一阶段正式作品
- 可以进入“打磨、优化、包装、准备面试讲解”的阶段
- 后续新增功能转入 `V1+` 或 `V2`
